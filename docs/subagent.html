Subagent 插件说明

本插件把一个委派任务交给独立启动的 pi 子进程执行。它隔离对话上下文，但不会隔离文件系统或项目目录；主代理随后得到子代理的最终输出。

插件：extensions/subagent/ 已检查 Pi：0.81.1 入口：index.ts 定义读取：agents.ts

先说结论

1

有四个内置角色

scout、planner、reviewer、worker。它们是 Markdown 文件中的角色提示词与配置，不是四个不同的插件。

2

有三种调用模式

单个执行（single）、并行执行（parallel）、串行交接（chain）。每次调用必须且只能选择其中一种。

3

可以选模型

可以。模型由“角色定义”的 YAML model 字段选择；当前工具调用参数中没有临时覆盖模型的 model 字段。

当前仓库中的角色

仓库的 agents/*.md 是安装源；执行 ./install.sh 后会复制到 ~/.pi/agent/agents/（或 PI_CODING_AGENT_DIR/agents/），插件运行时从那里重新发现角色。

角色 职责 当前模型 显式工具限制
scout 快速勘察代码库，产出可交接的压缩上下文、关键代码与入口。 openai-codex/gpt-5.6-luna read, grep, find, ls, bash
planner 基于需求和勘察结果输出实施计划；提示词要求只读、不修改。 openai-codex/gpt-5.6-sol read, grep, find, ls
reviewer 审查质量、安全性和可维护性；提示词要求 Bash 仅用于只读 Git 命令。 openai-codex/gpt-5.6-sol read, grep, find, ls, bash
worker 通用执行者，完成委派工作并报告变更文件和注意事项。 openai-codex/gpt-5.6-sol 未指定；子进程使用其 Pi 默认可用工具集。

“只读”是角色提示词和工具配置约束，不是操作系统沙箱。尤其不要把不可信项目中的项目级角色当作安全边界。

三种调用模式

Single：一个角色，一个任务

{
  "agent": "scout",
  "task": "找出认证相关代码"
}

适合一次性检索、审查或独立实现。

Parallel：多个任务并行

{
  "tasks": [
    { "agent": "scout", "task": "查模型层" },
    { "agent": "scout", "task": "查鉴权层" }
  ]
}

最多 8 个任务，同时最多运行 4 个子进程；结果按任务汇总。

Chain：逐步交接

{
  "chain": [
    { "agent": "scout", "task": "研究 X" },
    { "agent": "planner", "task": "基于以下结果制订计划：\n{previous}" }
  ]
}

每一步的最终文本替换下一步任务中的所有 {previous}；任一步失败即停止。

并行写入警告：并行任务可以使用相同的工作目录。若它们都会改同一文件，插件不会替你做跨子进程的冲突协调；并行模式更适合只读勘察或互不重叠的任务。

  1. 发现角色。discoverAgents() 读取用户级和/或最近父目录中的项目级 .pi/agents/*.md；缺少 name 或 description 的文件会被忽略。
  2. 选择唯一模式。提供 single、parallel、chain 中的一个。角色名不存在时，返回可用角色列表。
  3. 处理项目级信任。若请求 agentScope: "project" 或 "both"，且实际选择到项目级角色，交互界面默认弹出确认框。
  4. 构造子进程。基础参数为 --mode json -p --no-session；有模型时附加 --model，有工具列表时附加 --tools。角色系统提示词先写入权限为 0600 的临时文件，再经 --append-system-prompt 加载。
  5. 运行与收集。子进程在 cwd 参数指定的目录运行，否则使用父代理的当前目录。插件逐行解析 stdout 的 JSON 事件，收集助手消息、工具结果、轮次、token、缓存用量、费用和实际模型。
  6. 返回与展示。返回最后一条助手文本；chain 将该文本传给下一步，parallel 为每个任务生成小结。并行时给主代理看的每个任务输出上限为 50 KiB，完整结果仍保存在工具详情中。

按 Ctrl+O 可展开 TUI 中的子代理结果：查看完整任务、工具调用、Markdown 最终输出及各任务的 token/费用/模型统计。折叠状态会只显示最近项目（single 最多 10 条，chain/parallel 每项最多 5 条）。

上下文、文件与取消

真正隔离的内容

  • 子代理是另一个 pi 进程，并以 --no-session 启动。
  • 它不会继承主对话的完整上下文；只有任务文本、角色提示词和该子进程自己的 Pi 配置进入上下文。
  • chain 是显式交接：仅上一步“最后一条助手文本”会替换 {previous}。

没有隔离的内容

  • 工作目录、文件系统、Git 工作区、环境权限仍然由本机 Pi 子进程共享。
  • 角色的 cwd 可逐任务覆盖默认目录。
  • 用户级 Pi 配置、认证和已安装扩展会影响子进程的可用能力。

主代理取消时，AbortSignal 会发送 SIGTERM 给子进程；5 秒后如仍未结束则发送 SIGKILL。非零退出码、stopReason: error 或 aborted 都会被视为失败；chain 在失败步骤处终止。

角色来源与安全边界

作用域 读取位置 启用方式 覆盖规则
user（默认） ~/.pi/agent/agents/*.md 无需额外参数 只读取用户级角色
project 从当前目录向上找最近的 .pi/agents/*.md agentScope: "project" 只读取项目级角色
both 两处都读取 agentScope: "both" 同名项目级角色覆盖用户级角色
重要：项目级角色是仓库控制的提示词。交互模式下默认会确认；但 JSON/print 等无 UI 模式不会弹出确认框。因此只应对可信仓库使用 project/both，不要把 confirmProjectAgents 当成无 UI 运行时的安全防线。

模型：能否选择，以及怎样选择

能选择。当前实现的选择粒度是角色级，不是某一次工具调用级。插件读取角色 Markdown 的 YAML frontmatter： 

---
name: scout
description: Fast codebase recon
tools: read, grep, find, ls, bash
model: openai-codex/gpt-5.6-luna
---

这里是该角色附加给子 Pi 的系统提示词。
  • 存在 model 时：子进程会接收 --model <该值>。例如当前 scout 使用 Luna，其余三个内置角色使用 Sol。
  • 没有 model 时：插件不传 --model，子进程按自己的 Pi 默认模型配置运行；当前仓库的默认值是 openai-codex/gpt-5.6-sol。
  • 当前 subagent 参数 schema 没有 model 字段。因此不能在一次调用中写 { agent: "scout", model: "..." } 来覆盖 scout 的模型。
  • 模型名必须是当前 Pi 可识别且已具备认证的模型。仓库的 model-overrides.json 目前包含 gpt-5.6-luna、gpt-5.6-sol、gpt-5.6-terra 的覆盖配置。

修改现有角色的模型

# 编辑仓库源文件
# agents/scout.md
model: openai-codex/gpt-5.6-terra

# 安装到实际 Pi 配置目录
./install.sh

角色会在每次 subagent 调用时重新发现；安装后的角色文件修改会在下一次调用中生效。若修改的是仓库源文件，先执行安装脚本复制到实际 Pi 配置目录。模型或凭据不可用时，子进程会失败，错误会返回给主代理。

新增一个固定模型的角色

---
name: deep-reviewer
description: Uses Terra for a deeper code review
tools: read, grep, find, ls, bash
model: openai-codex/gpt-5.6-terra
---

只读检查变更；按严重性报告问题和路径、行号。

将其放到用户级 agents/deep-reviewer.md 并安装，之后以 agent: "deep-reviewer" 调用即可。也可放到可信项目的 .pi/agents/ 中，并显式使用 agentScope: "project" 或 "both"。

如果需要“每次调用临时指定模型”：当前插件尚未提供这一能力。需要扩展工具 schema（增加可选 model）并在 runSingleAgent() 中让该值优先于 agent.model；这会改变插件接口，不能仅通过调用参数实现。

预置工作流模板

prompts/ 中的模板不是新的子代理类型；它们只是指导主代理用 chain 模式调用前述角色。

命令 链路 用途
/scout-and-plan <任务> scout → planner 只调研和规划，不实施。
/implement <任务> scout → planner → worker 先建立上下文和计划，再实现。
/implement-and-review <任务> worker → reviewer → worker 实现、审查、根据反馈修正。

常见问题

为什么提示 Unknown agent？

检查角色文件是否已安装到实际 agent 目录、YAML 是否有非空 name 与 description、调用名是否一致，以及项目角色是否设置了正确的 agentScope。

为什么子代理没有使用当前主代理的模型？

它是新的 Pi 进程。模型优先来自角色的 model 字段；没有该字段才读取子进程自己的默认 Pi 配置。主会话当前临时选中的模型不会作为参数自动传递。

为什么 planner/reviewer 仍可能改文件？

它们的角色提示词要求只读，且 planner 不暴露 Bash、reviewer 的 Bash 被提示为只读；但这不是沙箱。应搭配可信模型、最小工具集和必要的外部权限控制。

为什么 chain 的下一步信息不完整？

只有前一步最后一条助手文本会传递。让前一步按结构化、压缩格式输出关键路径、代码片段和结论；需要保留原任务时，也应在后一步任务文本中显式写入。
